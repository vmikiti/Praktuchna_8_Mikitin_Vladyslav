## 1. Опис проекту
Розроблено консольну систему управління групою студентів. Програма дозволяє додавати студентів, вести журнал оцінок за предметами, автоматично розраховувати середній бал, фільтрувати успішних студентів та зберігати дані у формат JSON.

## 2. Реалізована архітектура
У проекті використано три основні класи:
*   **Student**: Модель студента з використанням інкапсуляції, `init-only` та `required` властивостей. Містить логіку перевірки ПІБ та номера заліковки.
*   **GradeJournal**: Клас для роботи з оцінками, реалізований через `Dictionary<string, double>`.
*   **StudentGroup**: Клас для управління списком студентів, реалізації пагінації та серіалізації даних.

## 3. Робота з Git
Процес розробки відбувався з використанням системи контролю версій Git:
*   **Ініціалізація**: Виконано команду `git init` у корені проекту.
*   **Ігнорування файлів**: Створено `.gitignore` для виключення папок `bin/`, `obj/` та `.vs/`.
*   **Гілки (Branching)**:
    *   `main` — стабільна версія проекту.
    *   `feature/student-class` — розробка моделі студента та журналу оцінок.
    *   `feature/group-management` — реалізація логіки управління групою.
    *   `feature/file-storage` — додавання функцій збереження в JSON.
*   **Фіксація змін**: Кожен етап завершувався окремим комітом з описовим повідомленням (наприклад: `Add validation for RecordBookNumber`).

## 4. Відповіді на контрольні питання

### 1. Що таке інкапсуляція? Приклади з коду.
**Інкапсуляція** — це приховування внутрішньої реалізації об'єкта та надання доступу до даних лише через контрольовані властивості.
**Приклади:**
1. Приватні поля `_fullName` та `_recordBookNumber` у класі `Student`.
2. Властивість `FullName` з перевіркою довжини в блоці `set`.
3. Приватний словник `_subjectGrades` у класі `GradeJournal`.
4. Властивість `AverageGrade` (тільки `get`).
5. Приватний список `_students` у класі `StudentGroup`.

### 2. Авто-властивості vs Повна реалізація.
*   **Авто-властивості** (`PersonalEmail`, `Status`): Використовувалися для простих даних без додаткової логіки.
*   **Повна реалізація** (`FullName`, `RecordBookNumber`): Обиралися для впровадження валідації та захисту даних від некоректного вводу.

### 3. Валідація даних.
Валідація застосована для ПІБ (не менше 5 символів), номера заліковки (рівно 8 цифр) та оцінок (від 0 до 100). Використовувалися логічні перевірки з викиданням `ArgumentException`.

### 4. Read-only та Init-only властивості.
*   **Read-only** (`Age`, `AverageGrade`): Дані, що обчислюються автоматично.
*   **Init-only** (`DateOfBirth`, `EnrollmentDate`): Дані, що встановлюються один раз при створенні та не можуть бути змінені пізніше.

### 5. Властивість Age.
Вона є обчислюваною, оскільки вік залежить від поточної дати. Це дозволяє завжди отримувати актуальне значення без необхідності оновлювати його вручну.

### 6. Принцип єдиної відповідальності (SRP).
Кожен клас має свою зону відповідальності: `Student` зберігає дані, `GradeJournal` рахує бали, `StudentGroup` маніпулює списком. Це робить код легким для підтримки.

### 7. Труднощі та рішення.
Виникали проблеми з видимістю класів через різні `namespaces`. Рішення: уніфікація всіх класів під один простір імен `Praktychna1`.
