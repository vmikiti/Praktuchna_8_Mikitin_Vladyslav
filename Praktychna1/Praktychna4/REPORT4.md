## 2. Відповіді на завдання для звіту

### 2.1. Навіщо потрібне перевантаження операторів?
Перевантаження операторів дозволяє використовувати стандартні математичні та логічні символи для об'єктів власних класів. Це підвищує читабельність коду, роблячи його ближчим до природної математичної мови. У проєкті це дозволило порівнювати студентів за успішністю безпосередньо через `> / <` та об'єднувати групи через `+`.

### 2.2. Застосування перевантаження у програмі
* **Клас Student:**
    * `>` та `<`: Порівняння за прогресом навчання (`CourseProgress`).
    * `==` та `!=`: Перевірка унікальності студента за номером заліковки.
    * `+`: Конкатенація ПІБ двох студентів у спільний рядок.
* **Клас StudentGroup:**
    * `+`: Об'єднання двох об'єктів груп в одну нову групу з усіма студентами.
* **Класи Complex та Vector:** Реалізовано арифметику для демонстрації роботи з числовими структурами.

### 2.3. Структура GradePoint та приведення типів
Замість простого `double` для оцінок використано структуру `GradePoint`:
1. **Implicit (неявне):** Дозволяє автоматично конвертувати `double` в `GradePoint` (зручно при ініціалізації).
2. **Explicit (явне):** Використовується для отримання числового значення, підкреслюючи свідомий вибір розробника.
3. **Operator true/false:** Використовується для швидкої перевірки оцінки (наприклад, оцінка >= 8 вважається істинною/відмінною).

### 2.4. Централізація розрахунків (Average Lab Grade)
Було реалізовано логіку, де `StudentGroup` виступає центральним вузлом обчислень:
* Метод `GetAverageLabGrade()` у студента тепер фільтрує нульові бали, надаючи реальну картину успішності.
* Властивість `AverageGroupLabGrade` у групі використовує LINQ (`Average`) для розрахунку середнього показника серед усіх студентів. Це позбавляє `Program.cs` від зайвих циклів та обчислень.

### 2.5. Реалізація індивідуального варіанту (Complex)
Створено клас `Complex` для роботи з комплексними числами $a + bi$.
* Реалізовано перевантаження `+`, `-`, `*`, `/`.
* Додано обчислення модуля числа через явне приведення до `double` за формулою $\sqrt{a^2 + b^2}$.

2.6. Труднощі та їх вирішення
Основна складність — необхідність перевизначати методи Equals() та GetHashCode() при перевантаженні ==. 
Якщо цього не зробити, колекції (як HashSet або Dictionary) працюватимуть некоректно. 
Ми вирішили це, пов'язавши логіку Equals безпосередньо з оператором ==.

2.7. Покращення читабельності
Замість громіздких методів myGroup.Merge(otherGroup) або if(s1.IsBetterThan(s2)) ми пишемо myGroup + otherGroup та if(s1 > s2). 
Це робить код схожим на математичне рівняння, що легше сприймається людським оком.

## 3. Контрольні питання

1. **Які оператори неможливо перевантажити?** Оператори присвоєння (`=`), доступу до членів (`.`), умовний оператор (`?:`), `new`, `typeof`, `sizeof`.
2. **Вимоги до сигнатури:** Методи операторів мають бути `public static`.
3. **True/False оператори:** Використовуються для того, щоб об'єкт можна було використовувати як умову в `if` або `while`.
4. **Стек vs Купа:** Структури (`struct`) зазвичай зберігаються у стеку (швидше), класи (`class`) — у купі.
5. Унарні vs Бінарні: Унарні приймають один аргумент (наприклад, !), бінарні — два (лівий і правий операнди).
6. Оператор приведення типу: Реалізується через public static [implicit/explicit] operator Type(SourceType value). Різниця в тому, чи змушуєте ви програміста писати назву типу в дужках.
7. Доцільність: Перевантаження варто використовувати лише тоді, коли дія оператора є інтуїтивно зрозумілою (наприклад, + для чисел або векторів). Якщо дія неочевидна (наприклад, Student * 5), краще використати звичайний метод із назвою.
