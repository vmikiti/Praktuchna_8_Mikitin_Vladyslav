1.Завдання для звіту.

1. Ключові відмінності між class і struct
Тип: Клас — посилальний тип (Reference Type), структура — тип значення (Value Type).
Пам'ять: Класи зберігаються в купі (Heap), структури — у стеку (Stack) або всередині об'єкта-контейнера.
Копіювання: При присвоєнні класу копіюється посилання, при присвоєнні структури — усі її дані (побітове копіювання).
Спадкування: Структури не підтримують успадкування (тільки реалізацію інтерфейсів).

2. Коли доцільно використовувати структури, а коли класи?
Структури: Для невеликих, незмінних об'єктів (до 16 байт), які представляють одне значення (точки Point, координати, грошові суми).
Класи: Для складних сутностей з поведінкою, станом та ієрархією (Student, StudentGroup).

3. Що таке readonly struct і які переваги він дає?
Це структура, де кожне поле позначене як readonly.
Переваги: Гарантує незмінність (immutability) даних, що робить код передбачуваним. Дозволяє компілятору уникати «захисних копій» при передачі в методи, значно підвищуючи швидкість.

4. Приклади реалізації у ваших структурах
У моєму проєкті для структури Point реалізація виглядає так:
C#
public readonly struct Point : IEquatable<Point>
{
    public int X { get; init; }
    public int Y { get; init; }

    public bool Equals(Point other) => X == other.X && Y == other.Y;
    public override bool Equals(object obj) => obj is Point other && Equals(other);
    public override int GetHashCode() => HashCode.Combine(X, Y);
}

5. Результати порівняння продуктивності
Результати: Масиви структур (StudentRecord[]) ініціалізуються та ітеруються швидше за масиви об'єктів.
Аналіз: Це пояснюється локальністю даних: структури лежать у пам'яті одним щільним блоком, що дозволяє процесору ефективніше використовувати кеш і не навантажувати Garbage Collector (GC).

6. Як інтеграція структур вплинула на систему?
Система стала більш продуктивною. Використання GradeRecord та StudentRecord дозволило зменшити споживання оперативної пам'яті та пришвидшити обробку великих списків оцінок, оскільки ці дані тепер не розкидані по всій купі.

7. Які труднощі виникали?
Основним викликом було розуміння механіки копіювання: наприклад, неможливість змінити поле структури безпосередньо в списку List<Point> (потрібно копіювати, змінювати і замінювати елемент повністю). Також вимагало уваги уникнення випадкового boxing (пакування).

8. Що нового ви дізналися?
Опанував практичне використання IEquatable<T> для оптимізації порівнянь, навчився працювати з незмінними структурами через readonly та init та зрозумів, як фізичне розташування даних у пам'яті впливає на швидкість роботи програми.

2. Контрольні питання.

1. Чому немає успадкування?
Щоб структура мала фіксований розмір у пам'яті та уникала проблеми «зрізання» даних.

2. При присвоєнні:
Всі поля структури копіюються в нову область пам'яті (створюється незалежний дублікат).

3. Навіщо GetHashCode?
Для коректної роботи в хеш-таблицях (Dictionary, HashSet) разом із Equals.

4. Перевага IEquatable: 
Усуває boxing (пакування в купу) та перевірку типів у рантаймі, що значно пришвидшує порівняння.

5. Коли виникає boxing?
При приведенні структури до типу object, інтерфейсу або використанні базового Equals(object).

6. Чому readonly struct краще? 
Вона безпечніша (незмінна) і дозволяє компілятору застосовувати агресивні оптимізації.

7. Як зменшують навантаження на GC? 
Структури зазвичай живуть у стеку або всередині масиву. Вони знищуються автоматично, не вимагаючи роботи прибиральника сміття.
